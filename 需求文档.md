## 一、项目背景

随着学习、工作和生活任务的增多，用户需要一个简洁、高效的任务管理工具，用于记录任务、规划时间、跟踪进度和查看完成情况。

 

你需要使用 Vue 技术栈完成一个功能完整、界面清晰、数据可保存的任务管理系统。系统应具备任务新增、编辑、删除、状态切换、搜索筛选和数据持久化等功能。



## 二、技术要求

### 1. 必选技术

- Vue 3
- Vite
- JavaScript 或 TypeScript
- Vue Router
- HTML5
- CSS3
- `localStorage`或数据库交互方案

### 2. 推荐技术

- Pinia
- Element Plus、Naive UI、Ant Design Vue 或其他 Vue UI 组件库
- ECharts
- Day.js
- VueUse

- Node.js + Express
- MySQL、SQLite、MongoDB 或云数据库

### 3. 开发工具

- VS Code
- Node.js
- npm、pnpm 或 yarn
- Chrome / Edge 浏览器
- Git

## 三、功能需求

### 1. 任务管理功能

系统应支持用户对任务进行基本管理：



- 新增任务
- 编辑任务
- 删除任务
- 查看任务列表
- 查看任务详情
- 批量清除已完成任务，可选

每个任务至少应包含以下字段：



| 字段        | 说明                       |
| ----------- | -------------------------- |
| 任务标题    | 必填，简要描述任务内容     |
| 任务描述    | 可选，记录任务详细说明     |
| 任务状态    | 待办、进行中、已完成       |
| 优先级      | 低、中、高                 |
| 截止日期    | 用于提醒和排序             |
| 分类 / 标签 | 如学习、生活、工作、项目等 |
| 创建时间    | 自动生成                   |
| 更新时间    | 编辑后自动更新             |

### 2. 任务状态功能

系统应支持任务状态切换：



- 待办
- 进行中
- 已完成

任务状态变化后，页面应立即更新对应显示效果。
建议设计不同状态的视觉样式，例如：



- 待办任务使用普通样式。
- 进行中任务使用醒目标签。
- 已完成任务使用删除线或浅色样式。

### 3. 搜索与筛选功能

系统应支持用户快速查找任务：



- 按关键词搜索任务标题或描述。
- 按任务状态筛选。
- 按优先级筛选。
- 按分类或标签筛选。
- 按是否逾期筛选，可选。

搜索与筛选结果应能实时更新。



### 4. 排序功能

系统应至少支持一种排序方式：



- 按创建时间排序
- 按截止日期排序
- 按优先级排序

优秀项目建议支持多种排序方式，并允许用户切换升序或降序。



### 5. 数据统计功能

系统应提供任务统计面板，展示任务完成情况。

 

基础统计项包括：



- 总任务数
- 待办任务数
- 进行中任务数
- 已完成任务数
- 任务完成率
- 逾期任务数

进阶统计项包括：



- 不同分类任务数量
- 不同优先级任务数量
- 最近 7 天任务完成趋势
- 使用 ECharts 绘制统计图表

### 6. 数据持久化功能

系统必须实现数据持久化。学生可根据自身能力选择以下两种方案之一：



#### 方案一：localStorage 本地存储

- 新增、编辑、删除任务后，数据应自动保存。
- 刷新页面后，任务数据不能丢失。
- 可提供“清空全部数据”功能。
- 可选实现数据导入、导出功能。

该方案适合基础版项目，不要求搭建后端服务。



#### 方案二：数据库交互

学生也可以采用前后端分离方式，通过接口与数据库交互，实现任务数据的持久化保存。

 

推荐实现方式：



- 前端使用 Vue 发送 HTTP 请求。
- 后端使用 Node.js + Express 提供接口。
- 数据库可选择 MySQL、SQLite、MongoDB 或云数据库。
- 前端通过 Axios 或 fetch 调用后端接口。

至少需要实现以下接口：



| 接口功能     | 请求方式    | 示例路径           |
| ------------ | ----------- | ------------------ |
| 获取任务列表 | GET         | `/api/tasks`       |
| 新增任务     | POST        | `/api/tasks`       |
| 修改任务     | PUT / PATCH | `/api/tasks/:id`   |
| 删除任务     | DELETE      | `/api/tasks/:id`   |
| 获取任务统计 | GET         | `/api/tasks/stats` |

数据库中的任务表或集合至少应包含：



| 字段        | 说明         |
| ----------- | ------------ |
| id          | 任务唯一标识 |
| title       | 任务标题     |
| description | 任务描述     |
| status      | 任务状态     |
| priority    | 任务优先级   |
| category    | 分类或标签   |
| dueDate     | 截止日期     |
| createdAt   | 创建时间     |
| updatedAt   | 更新时间     |

选择数据库方案的项目，应在 README 中说明后端启动方式、数据库连接方式和接口说明。



### 7. 页面与路由要求

项目至少包含以下页面：



| 页面              | 说明                               |
| ----------------- | ---------------------------------- |
| 首页 / 仪表盘     | 展示任务统计和近期任务             |
| 任务列表页        | 展示全部任务，支持搜索、筛选和排序 |
| 新增 / 编辑任务页 | 用于创建或修改任务                 |
| 统计分析页        | 展示任务统计数据                   |
| 设置页            | 用于主题切换、数据清空等，可选     |

**Tip**: 新增和编辑任务也设计为弹窗形式



## 四、界面要求

项目界面应满足以下要求：



1. 页面结构清晰，功能入口明确。
2. 操作按钮和表单控件布局合理。
3. 任务状态、优先级、截止日期等信息显示直观。
4. 页面整体风格统一。
5. 支持基本的响应式布局，能够适配电脑端和手机端。
6. 重要操作应有提示，例如删除确认、表单校验提示等。

## 七、难度分层要求

### 1. 基础必做

所有学生必须完成以下功能：



- 新增、编辑、删除任务
- 任务状态切换
- 搜索和筛选任务
- localStorage 或数据库持久化
- 基本页面样式

基础必做功能说明：



- 新增任务：用户可以创建一条新任务。
- 编辑任务：用户可以修改任务标题、描述、状态、优先级、截止日期等信息。
- 删除任务：用户可以删除不需要的任务。
- 状态切换：任务应支持待办、进行中、已完成等状态。
- 搜索筛选：用户可以按关键词、状态、优先级或分类查找任务。
- 数据持久化：可以使用 localStorage，也可以使用数据库接口保存任务数据。

### 2. 标准版要求

在基础版基础上增加：



- Vue Router 多页面结构
- Pinia 状态管理
- 优先级和分类管理
- 截止日期
- 排序功能
- 表单校验
- `Axios` 或 `fetch`接口请求封装，可选

### 3. 开放拓展

完成基础必做功能后，可以根据自己的兴趣和项目定位继续拓展。拓展功能不限制具体形式，可以围绕任务管理、时间规划、数据分析、用户体验、前后端交互等方向设计。

 

以下方向仅供参考，不要求全部完成，也可以自行设计其他合理功能：



- 看板视图：将任务按状态分为待办、进行中、已完成等列。
- 日历视图：按日期展示任务，方便查看每日任务安排。
- 统计图表：使用 ECharts 或其他图表库展示任务完成率、分类占比等数据。
- 深色 / 浅色主题切换：允许用户切换不同视觉主题。
- 拖拽排序：用户可以通过拖拽调整任务顺序或任务状态。
- 数据导入 / 导出：支持将任务数据导出为 JSON、CSV 等格式，也可以导入已有数据。
- 番茄钟或倒计时：帮助用户专注完成任务。
- 任务提醒：根据截止日期或提醒时间给出提示。
- 子任务：将一个大任务拆分为多个小步骤。
- 回收站：删除任务后可恢复，避免误删。
- 用户登录：区分不同用户的任务数据。
- 云端同步：通过后端接口或云数据库保存任务数据。

自拟拓展功能应满足以下要求：



- 与任务管理系统有明确关系。
- 能在项目中正常使用。
- README 中说明功能用途和实现思路。
- Git 提交记录中能体现该功能的开发过程。

## 八、Git 项目管理要求

本项目必须使用 Git 进行全过程管理。学生需要通过 Git 提交记录体现项目从创建、开发、完善到最终完成的推进过程。



### 1. 基本要求

- 项目开始时必须执行 `git init`，或创建远程 Git 仓库后克隆到本地开发。
- 每完成一个明确需求，必须进行一次 Git commit。
- 不允许所有功能完成后只提交一次 commit。
- 不允许只在最后一天集中补交大量无意义 commit。
- commit 记录应能体现真实开发过程。

### 2. Commit 规范

每次 commit 应只对应一个相对独立的功能、修复或优化。

 

推荐 commit message 使用以下格式：



```
类型: 简要说明本次提交内容
```

常用类型示例：



| 类型     | 说明       | 示例                        |
| -------- | ---------- | --------------------------- |
| init     | 项目初始化 | init: 创建 Vue 项目         |
| feat     | 新增功能   | feat: 实现任务新增功能      |
| fix      | 修复问题   | fix: 修复刷新后任务丢失问题 |
| style    | 样式调整   | style: 优化任务卡片样式     |
| refactor | 代码重构   | refactor: 拆分任务表单组件  |
| docs     | 文档修改   | docs: 补充 README 运行说明  |

### 3. Commit 示例

合理的提交过程示例：



```
init: 创建 Vue 3 项目
feat: 配置 Vue Router 页面路由
feat: 实现任务搜索和筛选
feat: 实现任务统计面板
style: 优化Web端页面布局
docs: 完善项目 README
```

### 4. Git 验收方式

验收时可以检查：



- 是否存在 `.git` 版本管理记录。
- 是否有清晰的 commit 历史。
- 是否基本做到“完成一个需求，提交一次 commit”。
- commit message 是否能说明本次提交内容。
- README 中是否提供 Git 仓库地址或提交记录查看方式。

可使用以下命令查看提交记录：



```
git log --oneline --graph --decorate
```

学生需提交以下内容：



1. Git 仓库地址，或包含 `.git` 目录的项目源码压缩包。
2. 项目运行说明文档 README.md，必须放在项目源码根目录中。
3. 简短项目总结，说明完成的功能、遇到的问题和解决方法。
4. 在 README 中说明 Git 提交记录查看方式。

README.md 至少应包含：



- 项目名称
- 项目简介
- 技术栈
- 功能列表
- 项目运行步骤
- Git 提交记录说明
- 数据持久化方案说明；如使用数据库，应包含后端启动方式、数据库配置方式和接口列表

## 十、提交前自查清单

提交项目前，请确认以下内容：



1. 项目能否正常安装依赖并运行。
2. 页面是否能正常访问。
3. 任务新增、编辑、删除是否有效。
4. 任务状态切换是否有效。
5. 搜索、筛选、排序是否有效。
6. 刷新页面后数据是否仍然存在；如使用数据库方案，请确认重新启动前后端后数据是否仍然存在。
7. 统计数据是否正确。
8. 代码是否存在明显报错。
9. Git 提交历史是否清晰，是否体现按需求推进项目。

## 十一、注意事项

1. 不要求必须连接真实后端，允许使用 `localStorage`；如果选择数据库方案，则必须提供可运行的后端接口和数据库说明。
2. 页面样式可以使用 UI 组件库，但不能只有组件堆砌，应体现自己的页面组织和交互设计。
3. 项目功能应稳定可用，避免只完成静态页面。
4. 提交前应删除无用文件，确保项目可以正常运行。
5. Git 提交记录属于提交检查内容，不得删除或伪造提交历史。
